## 2. 项目总定位

**Zhi Browser** 是一个面向个人长期自用的 Windows 桌面 AI 浏览器。

它不是普通 WebView 壳，也不是为了复刻 Chrome 的完整商业浏览器，而是一个围绕个人工作流、AI 阅读、网页增强、资源处理和高度可控界面构建的个人浏览器系统。

核心方向：

- 中文化优先
- 深色 UI
- 多标签浏览
- 自定义浏览器 chrome
- 浏览器功能可控
- AI 与网页内容深度结合
- 历史、书签、下载、代理等浏览器数据逐步变成可被 AI 使用的个人资料库
- 支持阅读、翻译、资源嗅探、用户脚本、分屏、侧边栏网页等个人高频功能
- 通过 Electron 快速利用 Chromium 能力，而不是从零开发浏览器内核

Zhi Browser 更像一个“个人网页工作台”，而不是一个单纯打开网页的壳。

## 6. 为什么使用 WebContentsView

Zhi Browser 早期尝试过 Electron 的 `<webview>` 路线。

但是 `<webview>` 在多标签、弹窗、导航控制、页面事件、布局同步、长期可维护性等方面都出现了明显问题。

因此项目已经放弃旧 `<webview>` 路线。

当前路线是：

```text
Electron + BaseWindow + WebContentsView
```

这个路线的好处是：

- 浏览器 UI 和网页内容边界更清楚
- React 负责浏览器界面
- WebContentsView 负责网页承载
- 主进程更容易管理标签页和页面内容
- 更适合实现侧栏、分屏、弹窗拦截、网页动作和 AI 集成
- 更适合后续长期维护

后续开发不应再回到旧 `<webview>` 路线。

## 8. 3.0.0 新增功能详解

### 8.1 内置多线程下载器

3.0.0 新增浏览器自己的下载核心。

它不是为了完全替代 IDM、FDM、NDM 等专业下载器，而是让 Zhi Browser 自己具备默认下载能力。

主要能力：

- 默认多线程下载
- 大文件自动分片
- 服务器支持 Range 时启用多线程
- 不支持 Range 时自动回退单线程
- 暂停
- 恢复
- 取消
- 断点续传
- 下载进度通知
- 任务持久化
- 下载完成后自动合并分片
- 清理临时文件
- 与外部下载器共存

设计意义：

资源嗅探、外部下载器和内置下载器共同构成 Zhi Browser 的资源处理闭环。

---

### 8.2 内置代理：mihomo 内核

3.0.0 新增内置代理能力，使用 mihomo 作为代理核心。

它的意义是让浏览器开始拥有自己的网络控制层，而不是完全依赖系统代理或外部代理客户端。

当前定位：

- 可用
- 可控
- 可切换
- 不追求一次性做成完整 Clash 客户端

后续可以继续扩展：

- 节点管理
- 配置导入
- 规则切换
- 按站点代理
- 按标签页代理
- 代理状态显示
- 代理失败提醒

---

### 8.3 独立历史记录管理页

历史记录不再只是一个临时列表，而是成为浏览器内部的正式页面。

目标能力：

- 查看历史记录
- 搜索历史记录
- 删除历史记录
- 快速重新打开历史网页
- 为 AI 搜索历史提供数据基础

历史记录是浏览器日用体验的重要组成部分，也是后续个人知识浏览器的重要数据来源。

---

### 8.4 独立书签管理页

书签管理从简单收藏按钮升级为独立页面。

目标能力：

- 查看全部书签
- 搜索书签
- 打开收藏网页
- 删除书签
- 整理书签
- 为 AI 搜索书签提供数据基础

书签不只是收藏链接，也可以成为个人知识库的一部分。

后续可以继续扩展：

- 自动分类
- 自动摘要
- 根据当前网页推荐已有书签
- 从书签中找相关资料
- 对长期收藏内容做知识整理

---

### 8.5 简易密码管理器

3.0.0 增加简易密码管理器。

当前定位是基础可用，不替代 1Password、Bitwarden 等专业密码管理器。

目标能力：

- 保存常用网站账号密码
- 本地查看和管理
- 降低重复输入成本
- 为个人浏览器提供基础账号体验

后续安全加固方向：

- 本地加密保存
- 主密码保护
- 避免明文日志
- 避免在渲染进程暴露敏感数据
- 区分普通设置数据和敏感凭据数据

### 8.6 无痕模式

3.0.0 新增无痕模式。

目标：

- 临时访问网页不污染普通历史记录
- 临时登录账号不影响主会话
- 测试网页状态更方便
- 提供更干净的临时浏览环境

后续需要重点验证：

- 是否写入普通历史记录
- 是否复用普通 cookie
- 是否有独立 session
- 关闭后是否清理临时数据
- 下载行为是否有提示

- 视频

- 压缩包

资源嗅探与下载系统的关系：

## 15. 数据与安全原则

运行数据应保存在 Electron userData 目录，不应保存在项目根目录或 Git 仓库中。

典型运行数据包括：

- preferences.json
- session.json
- userscripts.json
- downloads.json
- history data
- bookmarks data
- password data
- translation cache
- downloader configuration
- proxy configuration
- sidebar panel state
- split view state
- button visibility settings

不应进入 Git 的内容：

- node_modules
- dist 安装包
- 本地代理配置
- Claude / Codex / Gemini 本地配置
- 用户真实 API Key
- 用户真实浏览数据
- 用户真实密码数据

---

## 16. API Key 与隐私注意

公开仓库前需要检查：

- 是否存在真实 API Key
- 是否存在本地代理配置
- 是否存在个人路径敏感信息
- 是否存在真实账号密码
- 是否存在打包进安装包的开发者会话
- 是否将用户数据误放进源码目录

常见真实 Key 风险格式：

```text
sk-
sk-proj-
sk-ant-
AIza
```

字段名本身不是泄露，例如 `apiKey`、`token`、`secret` 只是代码字段名。

真正危险的是把真实密钥值写进源码、Git 历史或安装包。

- GitHub Release 下载来源被 Windows 标记
- Electron 安装包体积较大

用户可以通过：

```text

```

长期减少提示的方式是购买代码签名证书。
